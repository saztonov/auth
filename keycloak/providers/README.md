# keycloak/providers — кастомные SPI

Сюда кладутся `.jar` кастомных провайдеров Keycloak. Каталог монтируется в контейнер как
`/opt/keycloak/providers` (read-only). Команда `start` (без `--optimized`) выполняет `kc.sh build`
при старте и подхватывает провайдеры — после добавления/обновления jar нужен **рестарт** контейнера:

```bash
docker compose -p keycloak up -d   # с рестартом keycloak
```

> `.jar` в git не коммитим (см. корневой `.gitignore` — при необходимости добавить `*.jar`).
> Храним исходники/сборку провайдера или ссылку на релиз, а бинарь кладём при деплое.

## bcrypt PasswordHashProvider (миграция паролей BillHub)

BillHub хранит пароли как **bcrypt** (`$2[aby]$12$…`). Keycloak 26 по умолчанию хэширует **argon2**
(non-FIPS) и bcrypt «из коробки» не проверяет. Чтобы перенести хэши без сброса паролей, нужен провайдер
хэширования с алгоритмом `bcrypt`.

Как это работает:
1. Провайдер `bcrypt` кладётся в этот каталог, Keycloak пересобирается (рестарт).
2. Realm **password policy оставляем по умолчанию (argon2)**. bcrypt нужен только чтобы *проверить*
   перенесённые креды.
3. При импорте пользователя credential описывается как `algorithm=bcrypt`, `hashIterations=12`,
   `value=<полный $2a$12$…>`.
4. При первом успешном входе Keycloak видит расхождение алгоритма с policy и **перехэширует пароль в
   argon2** (пишет в `keycloak_db`). После этого bcrypt для пользователя больше не нужен.
5. Когда все активные пользователи вошли хотя бы раз — провайдер можно убрать.

Требования к провайдеру:
- совместимость с **Keycloak 26.1 (Quarkus)** — community-версии часто отстают от мажора, возможно,
  придётся собрать свой SPI (`PasswordHashProvider` + `PasswordHashProviderFactory`, provider id `bcrypt`);
- **формат `secretData`/`credentialData` при импорте должен совпадать с `verify()` провайдера** —
  доказать на отдельном тестовом realm ДО массовой заливки (см. план миграции BillHub).

Файлы сборки провайдера (если собираем свой) держать в отдельном подкаталоге, напр.
`providers/bcrypt-spi/` (Maven/Gradle-проект), а собранный jar — рядом при деплое.
