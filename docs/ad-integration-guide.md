# Интеграция Keycloak (контур su10) с Active Directory — инструкция для админов компании

Документ для **сетевого отдела и администраторов Active Directory компании**. Описывает, что нужно
настроить на стороне компании, чтобы сотрудники входили в корпоративные порталы su10 (EstiMat,
BillHub и др.) доменными учётными записями. Realm — единый `su10`, одна LDAP-федерация обслуживает
все порталы.

**Как это работает.** На стороне поставщика развёрнут Keycloak (`auth.su10.ru`, публичный сервер).
Сотрудник вводит доменный логин и пароль на странице входа; Keycloak проверяет их в вашем AD по
защищённому каналу и **не хранит у себя пароли сотрудников**. Подрядчики в AD не заводятся — они
ведутся отдельно в Keycloak как local users.

Связь Keycloak ↔ контроллер домена — **только через site-to-site IPsec-туннель**, протокол **LDAPS
(636/TCP)**. Контроллер домена в интернет не публикуется.

```
Сотрудник ──HTTPS──> Keycloak (auth.su10.ru, публичный сервер)
                        │  LDAPS 636/TCP
                        ▼
                  site-to-site IPsec (IKEv2)
                        ▼
              Контроллер домена (внутренняя сеть компании)
```

---

## 0. Обмен данными перед началом (заполнить совместно)

| Параметр | Кто предоставляет | Значение |
|---|---|---|
| Публичный IP сервера Keycloak (наша сторона) | Поставщик | `<VPS_PUBLIC_IP>` |
| Публичный IP вашего VPN-шлюза | Компания | |
| Внутренние IP / FQDN контроллеров домена (DC) | Компания | |
| Подсети или точные `/32` адреса DC для туннеля | Компания | |
| Способ аутентификации IPsec (PSK или сертификаты) | Совместно | |
| PSK / сертификаты IPsec | Совместно, **защищённым каналом** | |
| Base DN домена | Компания | напр. `DC=corp,DC=ru` |
| Users DN (OU с сотрудниками) | Компания | |
| Groups DN (OU с группами) | Компания | |
| Login-атрибут | Компания | `sAMAccountName` или `userPrincipalName` |
| Bind DN сервисной учётки | Компания | напр. `CN=svc-keycloak-ldap,OU=Service,DC=corp,DC=ru` |
| Пароль сервисной учётки | Компания, **защищённым каналом** | |
| Публичная часть корпоративного CA (root/intermediate) | Компания | файл `.pem`/`.cer` |

> Все секреты (PSK, пароль bind-учётки, приватные ключи) передаются **только защищённым каналом**.
> **Приватные ключи сертификатов DC не передаются никогда** — нужна только публичная часть CA.

---

## 1. Сеть: site-to-site IPsec (IKEv2)

1. Поднять на корпоративном VPN-шлюзе **site-to-site IPsec, IKEv2** навстречу `<VPS_PUBLIC_IP>`.
2. Параметры (рекомендуемый набор): IKEv2; аутентификация PSK или сертификаты; шифрование
   **AES-256-GCM**, целостность/PRF **SHA-256**, DH-группа **14+** (19/20 на эллиптике при поддержке).
3. **Узкие traffic selectors:** в туннеле — только `<VPS_PUBLIC_IP>/32` ↔ конкретные DC (`<DC_IP>/32`).
   Всю внутреннюю сеть не заворачивать.
4. **Firewall:** от нашего сервера к DC — только **636/TCP (LDAPS)**. `3269/TCP` (GC over SSL) — только
   при поиске по глобальному каталогу. Остальное закрыть.
5. Настроить мониторинг/алерты доступности туннеля и LDAPS — простой канала = невозможность входа.

---

## 2. LDAPS на контроллерах домена

1. Включить **LDAPS** (нужен серверный сертификат на DC).
2. Сертификат DC: выпущен корпоративным/доверенным CA; в **SAN** — имя, по которому Keycloak
   обращается к DC (FQDN/IP); EKU **Server Authentication** (`1.3.6.1.5.1.5.7.3.1`).
3. Проверить, что DC слушает **636/TCP** (`ldp.exe` → SSL, 636).
4. Передать нам **публичную часть** CA (`.pem`/`.cer`). Приватные ключи — не передаются.

---

## 3. Сервисная учётная запись для LDAP bind

1. Создать учётку, напр. `svc-keycloak-ldap`.
2. Права — **минимальные, только чтение**: read к Users DN и Groups DN; запрет интерактивного входа;
   не включать в админ-группы; пароль не истекает либо согласованная ротация с уведомлением.
3. Keycloak работает в режиме **READ-ONLY**: пользователи и пароли остаются в AD.
4. Передать **Bind DN** и **пароль** (защищённым каналом).

---

## 4. Группы доступа и ролей в AD

Доступ к каждому порталу и роли определяются членством в группах AD. Realm `su10` общий, но группы
**отдельные на каждый портал** (сотрудник может иметь доступ к EstiMat, но не к BillHub, и наоборот).
Создайте группы (имена — пример) и сообщите их **полные DN**:

**EstiMat (сметный портал):**

| Группа AD (пример) | Назначение |
|---|---|
| `EstiMat-Access` | Общий доступ. **Без неё вход в EstiMat невозможен.** |
| `EstiMat-Admins` | Роль «Администратор» |
| `EstiMat-Engineers` | Роль «Инженер-сметчик» |
| `EstiMat-Managers` | Роль «Руководитель» |

**BillHub:**

| Группа AD (пример) | Назначение |
|---|---|
| `BillHub-Access` | Общий доступ к BillHub. |
| `BillHub-Admins` / `BillHub-Users` / `BillHub-Security` | Роли BillHub (по вашей ролевой модели) |

Правила:
- для входа в портал нужна как минимум его `*-Access`; роли — отдельными группами;
- роль «Подрядчик» сотрудникам компании **не** назначается (подрядчики — отдельно в Keycloak);
- членство кэшируется; изменения применяются при следующем входе/синхронизации.

> Примечание про роли: некоторые порталы (например BillHub) держат авторизацию в своей БД, а не в
> ролях Keycloak. Для них `*-Access` определяет право входа/идентичность, а конкретные роли/доступы
> портал берёт из своей базы. Согласуйте с поставщиком, какие группы к чему маппятся для каждого портала.

---

## 5. Учёт состояния учётных записей

- Отключённые/заблокированные в AD (`userAccountControl`) теряют доступ к порталам — стандартное
  отключение учётки при увольнении достаточно.
- Смена доменного пароля действует сразу — порталы отдельный пароль не хранят.

---

## 6. Граница ответственности (наша сторона)

- Поднимаем IPsec навстречу вашему шлюзу, проверяем LDAPS до DC.
- Импортируем ваш CA, настраиваем в Keycloak LDAPS-федерацию (READ-ONLY) с вашей сервисной учёткой.
- Маппим группы AD на роли/доступ порталов (раздел 4).
- Проверяем входы, мониторим bind/туннель со своей стороны.

---

## 7. Приёмочная проверка (совместно)

1. Туннель IPsec стабилен; с нашего сервера проходит LDAPS bind к DC.
2. Член `EstiMat-Access` входит в EstiMat; не-член — не входит. Аналогично `BillHub-Access` и BillHub.
3. Отключённая в AD учётка — входа не получает.
4. Членство в ролевых группах корректно даёт роль.

---

## Сводка: что прислать нашей стороне

1. Публичный IP VPN-шлюза, IP/FQDN DC, подсети/`/32` для туннеля.
2. Параметры IPsec и PSK/сертификаты (защищённым каналом).
3. Публичную часть корпоративного CA.
4. Bind DN и пароль `svc-keycloak-ldap` (защищённым каналом).
5. Base DN, Users DN, Groups DN, login-атрибут.
6. DN групп доступа/ролей по каждому порталу (`EstiMat-*`, `BillHub-*`).
